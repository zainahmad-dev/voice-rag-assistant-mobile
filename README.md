# Voice RAG Assistant — Mobile

Ask questions about your own documents — by voice or by text — and get answers grounded strictly in
what you uploaded. Upload a PDF, DOCX, or TXT file, wait for it to be indexed, then either type a
question or tap the mic and talk to it. Every answer is retrieved from your documents via vector
search and generated by an LLM instructed never to go beyond that retrieved context, with the source
document and similarity score shown alongside each reply.

This is the native iOS/Android client. It is a second front end for an existing product, not a new
system — see [Relationship to the web app](#relationship-to-the-web-app) below.

## Relationship to the web app

There are two clients and **one backend**:

```
┌─────────────────────────┐        ┌──────────────────────────┐
│  Web app (Next.js PWA)  │        │  Mobile app (this repo)  │
│  voice-rag-assistant    │        │  Expo React Native       │
└────────────┬────────────┘        └────────────┬─────────────┘
             │                                   │
             │   HTTPS: /api/upload, /api/ingest,│
             │   /api/documents, /api/query,     │
             │   /api/vapi/webhook               │
             └─────────────────┬─────────────────┘
                               ▼
        ┌──────────────────────────────────────────────┐
        │  Shared backend — one Vercel deployment       │
        │  https://voice-rag-assistant-blush.vercel.app │
        │  Supabase Postgres + pgvector + Storage       │
        └──────────────────────────────────────────────┘
```

The backend lives in the sibling repo [`voice-rag-assistant`](https://github.com/zainahmad-dev/voice-rag-assistant)
(the Next.js web app **is** the API). This mobile client ships no server code of its own — it calls the
same REST endpoints and the same VAPI tool webhook the web app uses. A document uploaded on the web
appears in the mobile library and can be answered by a mobile voice question, and vice versa; this was
verified end to end during development.

Because the two clients share a backend, they also deliberately share the **design language**: the
color tokens, typography roles, and spacing scale in `src/theme/` mirror the web app's `globals.css`,
so the two feel like one product.

## Features

- **Document upload** for PDF, DOCX, and TXT (15 MB max), with live per-document indexing status
  (pending → processing → completed / failed) via list polling.
- **Voice conversations** powered by VAPI — real-time speech-to-text, an LLM, and text-to-speech in one
  call, with the assistant calling back to the shared RAG webhook for grounded answers.
- **Text chat** as a lighter alternative to voice, sharing the same conversation history.
- **Retrieval-augmented answers**: every reply is grounded in your documents' most similar chunks, with
  source attribution (document name + similarity score).
- **Document library** with per-document status and deletion.
- **Dark / light mode** that follows the system theme until you override it, and persists your choice.
- **Offline-aware error handling** and a shared loading affordance across every async operation.
- **Persistent conversation** — transcripts survive app restarts (AsyncStorage).

## Tech-stack decisions and reasoning

**Expo dev client instead of Expo Go.** VAPI's React Native SDK depends on `@daily-co/react-native-daily-js`
and a native `react-native-webrtc` fork. Those are native modules that are **not** part of the Expo Go
runtime, so the app cannot run in Expo Go at all — it needs a custom development build (a "dev client")
that bundles those native modules. This is the single decision that shapes the whole workflow: there is
no "scan a QR code in Expo Go" path here; you install a dev-client build first (see below).

- **New Architecture is disabled** (`newArchEnabled: false` in `app.json`). Daily's WebRTC fork is a
  legacy-only module; it does not run under the New Architecture. This flag is load-bearing — do not
  flip it on.
- **`@daily-co/react-native-daily-js` is pinned to an exact version** (`0.78.0`), as are the WebRTC and
  AsyncStorage versions it needs (see the `overrides` in `package.json`). Bumping daily-js pulls in a
  newer WebRTC that breaks voice. Treat the voice dependency set as frozen.
- **No `react-native-reanimated`.** The one animation in the app (the pulsing voice orb) uses React
  Native's built-in `Animated` API. Reanimated was removed after a native-worklets crash; the built-in
  API is enough for this UI and avoids reintroducing that failure mode.

**Bottom tabs for navigation.** The app has two primary destinations — **Library** (manage documents)
and **Assistant** (ask questions). A bottom tab bar is the platform-standard, thumb-reachable pattern
for a small fixed set of top-level destinations, so it beats a drawer or stack here. React Navigation's
tab colors are wired to the active theme so the chrome matches the palette.

**Same design tokens as the web app.** `src/theme/colors.ts`, `typography.ts`, and `spacing.ts` port the
web app's design system verbatim (the dark palette is copied from the web `globals.css`, the font roles
map Space Grotesk / Inter / JetBrains Mono to display / body / data). Sharing tokens keeps the two
clients visually consistent for free and means design changes have one source of truth.

## Environment variables

Copy `.env.example` to `.env` and fill in the values:

| Variable | Purpose |
| --- | --- |
| `EXPO_PUBLIC_API_URL` | Base URL of the shared backend. Defaults to the deployed Vercel host. |
| `EXPO_PUBLIC_VAPI_PUBLIC_KEY` | VAPI **public** (client) key, from dashboard.vapi.ai → Organization → API Keys. |

Both are prefixed `EXPO_PUBLIC_`, which means Expo **inlines them into the app bundle at build time** —
they are readable by anyone with the app. That is fine for these two values (the VAPI public key can
only *start* calls, and the API URL is public), but it also means **no secret ever belongs in this
file.** The VAPI private key and all Supabase/LLM keys live only in the backend.

> Changing an `EXPO_PUBLIC_` value requires restarting the dev server (and rebuilding for a standalone
> build) because the value is baked in, not read at runtime.

## Local development

Prerequisites: **Node.js 22+**, an [Expo account](https://expo.dev), and the EAS CLI
(`npm install -g eas-cli`). For voice you also need a device or emulator with a microphone.

```bash
# 1. Install dependencies
npm install

# 2. Configure environment
cp .env.example .env        # then edit .env with your values

# 3. Build and install a dev client (ONE TIME, per platform).
#    Expo Go will not work — the app has native voice modules.
eas build --profile development --platform android   # or: ios
#    Install the resulting build on your device/emulator.

# 4. Start the Metro dev server and open it in the dev client
npx expo start --dev-client
```

After the one-time dev-client install, day-to-day work is just step 4 — edit code and it hot-reloads in
the installed dev client. You only need to rebuild the dev client when native dependencies or native
config (`app.json`, permissions, the New Architecture flag) change.

If you have the native toolchains installed locally (Android Studio / Xcode), you can build the dev
client on your own machine instead of on EAS with `npx expo run:android` / `npx expo run:ios`.

## EAS Build

Builds are produced by [EAS Build](https://docs.expo.dev/build/introduction/). Profiles are defined in
`eas.json`:

| Profile | Command | Output |
| --- | --- | --- |
| `development` | `eas build --profile development --platform android` | Dev client (Metro-connected), internal distribution |
| `preview` | `eas build --profile preview --platform android` | Standalone installable APK, internal distribution |
| `production` | `eas build --profile production --platform android` | Store-ready build, auto-incremented version |

```bash
# One-time setup
npm install -g eas-cli
eas login

# Produce an installable APK for testing/submission
eas build --profile preview --platform android
```

EAS prints a URL when the build finishes; download the APK from there (or install directly on a
connected device). The EAS project is `54b63d4b-19de-4878-8b2a-54f63ab6e04e` under the `zainahmaddev`
owner (see `app.json` → `extra.eas`). App identifiers are `com.zainahmaddev.voiceragassistantmobile`
for both platforms.

> **iOS note:** a `preview`/`production` iOS build (and running on a physical iPhone) requires an Apple
> Developer account for signing. Android APKs have no such requirement, which is why the submission
> deliverable is an Android APK.
